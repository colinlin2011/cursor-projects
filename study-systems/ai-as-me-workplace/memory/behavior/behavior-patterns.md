## 高效行为模式

### 模式1：及时止损模式
- **描述**：当发现方案不合适或投入产出比不佳时，及时放弃，不继续投入资源
- **适用场景**：技术方案评估、工具选择、资源投入决策
- **行为步骤**：
  1. 识别方案与需求不匹配
  2. 评估继续投入的成本
  3. 及时做出放弃决策
  4. 转向更合适的方案
- **有效性**：高（避免资源浪费，提高效率）
- **使用频率**：中
- **成功案例**：[DEC-001 放弃Docker方案](work/decisions/decisions/20260113-pandoc-docker-decision.md)

### 模式2：系统化调试模式
- **描述**：通过运行时证据（调试日志、运行时数据）定位问题，而不是仅基于代码分析猜测
- **适用场景**：Bug修复、功能调试、性能优化
- **行为步骤**：
  1. 生成3-5个精确假设
  2. 在代码中添加调试日志，测试所有假设
  3. 运行程序，收集运行时证据
  4. 分析日志，评估每个假设（确认/拒绝/不确定）
  5. 基于证据修复问题
  6. 验证修复效果
- **有效性**：高（基于证据修复，准确性高）
- **使用频率**：高（技术问题修复常用）
- **成功案例**：[故障诊断日志提取功能修复记录](../../work/fault_diagnosis_log_extraction_fix_20250115.md)

### 模式3：逻辑严格性模式
- **描述**：在实现多条件过滤逻辑时，确保条件严格，使用 `and` 而非 `or`，避免误提取
- **适用场景**：数据提取、过滤逻辑、条件判断
- **行为步骤**：
  1. 明确每个过滤条件的含义
  2. 使用真值表验证逻辑正确性
  3. 将复杂逻辑拆分为多个步骤
  4. 确保每个步骤的条件明确
  5. 添加调试日志验证逻辑执行路径
- **有效性**：高（避免逻辑错误，提高准确性）
- **使用频率**：中（数据处理场景常用）
- **成功案例**：[故障诊断日志提取功能修复记录](../../work/fault_diagnosis_log_extraction_fix_20250115.md)
